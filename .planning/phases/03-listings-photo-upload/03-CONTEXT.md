# Phase 3: Listings & Photo Upload - Context

**Gathered:** 2026-03-25  
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can publish items for sale with photos and geolocation. Listing creation includes address-to-coordinates conversion and secure multi-photo upload. Listing search (distance filtering) is Phase 4. Chat/messaging is Phase 5.

</domain>

<decisions>
## Implementation Decisions

### Photo Upload Flow
- Photos uploaded **after** listing created (user creates listing first, then uploads photos)
- Backend accepts **max 10 photos** per listing (409 Conflict if exceeded)
- Upload delivery: **Multipart form** with multiple files in single POST request
- Photo metadata stored: `id, listing_id, photo_url, display_order, created_at, deleted_at` (schema already exists)

### Listing Creation Form
- **Multi-step wizard** structure (not single form)
- **Required fields:** title, category, price, address, condition
- **Optional fields:** brand, model, year, accessories
- **Geocoding timing:** On form submit (validate & geocode address in POST /api/listings)
- **Geocoding failure handling:** Suggest alternatives immediately if address invalid/ambiguous
- Backend returns list of candidates; user confirms or corrects before creating listing

### Photo Storage & Display
- **Primary/cover photo:** First photo in display_order (reorderable)
- **Filename strategy:** `{timestamp}_{random}.{ext}` (e.g., `1726918234_a1b2c3.jpg`)
  - Provides both debuggability and security against enumeration attacks
- **Storage path:** `public/uploads/users/{user_id}/listings/{listing_id}/`
  - Organized by user then listing for clarity and isolation
- **Thumbnails:** Not generated by backend (frontend scales images via CSS)
- **Image re-encoding:** Strip EXIF/metadata before storage (security requirement from Phase 3 pitfalls)

### Address & Geolocation
- **Coordinate precision:** High (DECIMAL(10, 8) → ~1mm accuracy)
- **User visibility:** Address string only displayed (coordinates stored internally for search)
- **Manual correction allowed:** 
  - User can edit address and re-trigger geocoding
  - User can correct via map picker (if map interface provided later)
- **Geocoding API failure:** Suggest alternatives immediately; if no alternatives available, return 400 with message for user to retry/revise address
- **Soft delete on listings:** All listing_photos inherit soft delete when parent listing soft-deleted

### OpenCode's Discretion
- Exact order of wizard steps (1. basic info, 2. location, 3. photos, 4. review | other sequence)
- HTTP status codes for edge cases (e.g., max photos exceeded)
- Image re-encoding implementation details (library choice, quality settings)
- Geocoding API selection and fallback strategy
- Exact format/content of suggestion list when address is ambiguous

</decisions>

<specifics>
## Specific Ideas

- Backend should handle the complete validation pipeline: required fields check → address geocoding → photo count validation
- Photos should be reorderable via frontend before final submission (UX consideration for planner)
- Consider caching geocoding results to reduce API calls (addresses checked before, can reuse coordinates from geocoding_cache table)

</specifics>
