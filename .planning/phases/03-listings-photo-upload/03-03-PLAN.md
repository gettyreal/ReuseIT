**Step 1: Extend GeolocationService (from Phase 2) with new methods**

Add method: geocodeAddressWithCandidates($address): Returns array of candidates
- Normalize address using existing normalizeAddress() method
- Generate cache key (MD5 hash) and check geocoding_cache table for exact match
- If cache hit: return single candidate [{'address': cached_string, 'lat': cached_lat, 'lng': cached_lng}]
- If cache miss: Call Google Maps Geocoding API with normalized address
  - Parse response['results'][] array
  - Extract formatted_address, geometry.location.lat, geometry.location.lng from each result
  - If 0 results: Return null (address not found)
  - If 1 result: Cache and return single candidate
  - If multiple results (2-5): Return all as array for user to pick
- Store only FIRST result (highest confidence) in cache if multiple results exist
  - Log in cache: formatted_address, lat, lng, created_at

Handle errors gracefully:
- Google API error (rate limit, quota, malformed address): Return null or empty array (don't crash)
- Network timeout: Return null

**Step 2: Update ListingService.createListing()**

Change geocoding flow:
1. Call $geolocation->geocodeAddressWithCandidates($address)
2. If null returned: Return Response::error('Address not found. Please try again with exact address.', 400)
3. If 1 candidate: Auto-select, store lat/lng in listing
4. If multiple candidates: Return Response::success(['listing_candidate' => null, 'candidates' => $candidates]) with HTTP 200
   - Frontend receives candidates list for user to pick
   - User re-submits with confirmed address (or corrected)
   - On re-submit: Call geocodeAddressWithCandidates() again with corrected address

**Step 3: Handle address confirmation flow (optional, discretionary)**

Option A (simple): Require user to re-submit form with corrected address when ambiguous
Option B (advanced): Accept candidate_index parameter in second POST to select from previous candidates

Implement Option A for MVP (simpler, avoids state tracking).

**Step 1: Add filter methods to ListingRepository**

Add methods (all with soft-delete filtering via Softdeletable trait):

- searchByKeyword($keyword, $limit = 20, $offset = 0): array
  - Query listings WHERE (title LIKE ? OR description LIKE ?) AND deleted_at IS NULL
  - Use prepared statement: "%" . $keyword . "%"
  - Return array of listings ordered by created_at DESC with LIMIT and OFFSET
  - Include seller_id, title, description, price, condition, photo_url

- filterByCategory($category_id, $limit = 20, $offset = 0): array
  - Query listings WHERE category_id = ? AND deleted_at IS NULL
  - Return paginated results

- filterByCondition($condition, $limit = 20, $offset = 0): array
  - Query listings WHERE condition = ? AND deleted_at IS NULL
  - Validate condition enum: ['Excellent', 'Good', 'Fair', 'Poor']

- filterByPrice($min_price, $max_price, $limit = 20, $offset = 0): array
  - Query listings WHERE price BETWEEN ? AND ? AND deleted_at IS NULL
  - Validate min_price < max_price

- filterCombined($filters = [], $limit = 20, $offset = 0): array
  - Accept filter hash: {'category_id': 1, 'condition': 'Excellent', 'price_min': 100, 'price_max': 500, 'keyword': 'iphone'}
  - Build dynamic query with WHERE conditions for each provided filter
  - Combine with AND: WHERE category_id = ? AND condition = ? AND price BETWEEN ? AND ? AND (title LIKE ? OR description LIKE ?) AND deleted_at IS NULL
  - Return paginated results

All methods use JOIN to include photos (GROUP_CONCAT) and seller info (users.name, users.avg_rating) to avoid N+1.
