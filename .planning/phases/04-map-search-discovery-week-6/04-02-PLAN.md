## Tasks

### Task 1: Add searchWithDistance() Method to ListingService

**Name:** Implement distance-based search with fallback location and radius filtering

**Files:** `src/Service/ListingService.php` (MODIFY)

**Action:**

Add new method `searchWithDistance()` to ListingService:

1. **Method signature:**
   ```php
   public function searchWithDistance(
       ?float $userLat,
       ?float $userLng,
       int $radiusMeters,
       array $filters = [],
       int $limit = 20,
       int $offset = 0
   ): array
   ```

2. **Parameter handling:**
   - `$userLat`, `$userLng`: user's search location (nullable)
   - `$radiusMeters`: search radius in meters (will be validated)
   - `$filters`: array with optional keys (keyword, category_id, condition, price_min, price_max)
   - `$limit`, `$offset`: pagination parameters

3. **Validation logic (in this order):**
   - Check radius: if `$radiusMeters < 0 || $radiusMeters > 50000`, throw `InvalidArgumentException` with message "Radius must be between 0 and 50000 meters"
   - Check location not null: if both `$userLat` and `$userLng` are null, throw `InvalidArgumentException` with message "User location required for distance search"
   - Check coordinates valid:
     - Latitude: `-90 <= $userLat <= 90`
     - Longitude: `-180 <= $userLng <= 180`
     - If invalid, throw exception with descriptive message

4. **Search flow:**
   - **Step 1:** Call `$this->listingRepository->searchCandidatesByFilters($filters, $limit = 1000)`
   - **Step 2:** Fetch all candidates (up to 1000 results)
   - **Step 3:** For each candidate listing:
     - Skip if listing missing `latitude` or `longitude` (edge case, but safe)
     - Call `GeometryService::haversineDistance($userLat, $userLng, listing['latitude'], listing['longitude'])`
     - Convert result from km to meters: `$distanceMeters = $distanceKm * 1000`
     - Add field to listing: `$listing['distance_meters'] = $distanceMeters`
     - If `$distanceMeters <= $radiusMeters`, add listing to filtered set
   - **Step 4:** Sort filtered set by distance (nearest first):
     - Use `usort($filtered, fn($a, $b) => $a['distance_meters'] <=> $b['distance_meters'])`
   - **Step 5:** Apply pagination:
     - `$paginated = array_slice($filtered, $offset, $limit)`
   - **Step 6:** Return result array:
     ```php
     [
         'listings' => $paginated,
         'total' => count($filtered), // Total within radius
         'limit' => $limit,
         'offset' => $offset,
         'search_radius_meters' => $radiusMeters,
         'search_center' => ['latitude' => $userLat, 'longitude' => $userLng]
     ]
     ```

5. **Error handling:**
   - Catch any exceptions from `searchCandidatesByFilters()` and re-throw with context
   - If Haversine calculation fails (very unlikely), skip listing and continue
   - Empty result set is valid (return 200 with empty array, not error)

6. **Performance notes:**
   - Acceptable for MVP: up to ~10,000 candidates + Haversine sort = ~50-100ms
   - No database pagination at this layer (client receives full sorted result, then paginated)
   - If becomes bottleneck: Phase 8 migration to MySQL ST_Distance()

**Verify:**

- [ ] Method signature matches specification above
- [ ] Radius validation: rejects < 0 and > 50000
- [ ] Location validation: rejects null/invalid coordinates
- [ ] Distance calculation applied to all candidates
- [ ] Listings without lat/lng safely skipped
- [ ] Results sorted by distance (nearest first)
- [ ] Pagination applied after sorting
- [ ] Return format includes `total`, `limit`, `offset`, `search_radius_meters` metadata

**Done:**

- Call with valid user location and empty filters: returns listings within radius sorted by distance
- Call with keyword filter: only listings matching keyword returned
- Call with category + price filters: only matching listings returned
- Call with radius = 0: only listings at exact location or within ~1m (Haversine precision)
- Call with radius > 50000: throws exception with "50000 meters" message
- Pagination: offset = 20, limit = 20 returns results 20-39

### Task 2: Add Location Fallback to ListingService

**Name:** Implement user location fallback when search coordinates not provided

**Files:** `src/Service/ListingService.php` (MODIFY)

**Action:**

Create new method `getUserLocation()` or integrate into `searchWithDistance()`:

1. **When lat/lng are null in searchWithDistance():**
   - Get current user ID from session: `$userId = $_SESSION['user_id'] ?? null`
   - If no session user, throw exception: "Authentication required for default location search"
   - Call `$this->userRepository->find($userId)` to fetch user record
   - Check user's stored location: if `$user['latitude']` or `$user['longitude']` is null, throw exception: "User location not set in profile"
   - Use `$user['latitude']` and `$user['longitude']` as fallback

2. **Implementation approach:**
   - Add method `getUserLocation()` that returns `['latitude' => ..., 'longitude' => ...]`
   - Call from `searchWithDistance()` at start: if parameters null, use fallback
   - Or inline in `searchWithDistance()` for simplicity

3. **Integration with existing code:**
   - ListingService constructor already has `UserRepository` injected (from Phase 2)
   - Reuse existing injection (no new dependencies)

4. **Error handling:**
   - Catch `UserRepository::find()` exceptions and re-throw as `InvalidArgumentException`
   - Return descriptive message for user debugging

**Verify:**

- [ ] Location fallback called when lat/lng null
- [ ] User location fetched from database correctly
- [ ] Exception thrown if user location null
- [ ] Exception thrown if user not authenticated
- [ ] Existing UserRepository integration used (no new injection)
